# MVP-3.1b Anomalies - Perf Evidence

> Status: landed | Phase: MVP-3.1b | Companion to [docs/admin-v2-mvp-3-anomalies-first-slice.md](./admin-v2-mvp-3-anomalies-first-slice.md)

This is the perf-evidence doc that closes the placeholder left in slice
3.1a (`Performance proof` section: "still need to be attached from an
environment with the target database fixture set").

## Scope

Two things are proven here:

1. The read-only ledger anomaly surface
   (`/api/admin-v2/ledger/anomalies/{summary,list}`) stays inside the
   summary endpoint budget of **500 ms p95** at the synthetic scale used
   below.
2. The single new index added by migration
   `20260420170000_admin_v2_anomalies_outcome_indexes` (`idx_ledger_entry_outcome_lateral_covering`,
   covering INCLUDE, see
   [migration README](../packages/database-2/prisma/migrations/20260420170000_admin_v2_anomalies_outcome_indexes/README.md))
   is the right shape for the LATERAL pattern used by every anomaly
   query.

Anti-scope for this doc (owned by later slices):

- no new query shapes;
- no UI changes;
- no production capture (production volumes are too small today, see
  3.1a README);
- no decision about dropping the dormant 3.1a indexes
  (`ledger_entry(type, status)` and `ledger_entry(status, created_at)`),
  which is owned by the future "Reassessment threshold" trigger.

## Method

- Synthetic dataset generated by
  [`scripts/admin-v2-anomalies-perf/seed.mjs`](../scripts/admin-v2-anomalies-perf/seed.mjs):
  - 5,000 perf-namespaced consumers;
  - 50,000 `ledger_entry` rows distributed across them;
  - 124,551 `ledger_entry_outcome` rows (1-4 per entry, append-only
    chain);
  - deliberate injections so every anomaly class returns non-empty
    results: 200 stale-pending, 100 inconsistent-chain, 300 large-value
    outliers across the 11 tracked currencies.
  - `ANALYZE` is run on `ledger_entry` and `ledger_entry_outcome`
    immediately after insertion so planner stats reflect the synthetic
    distribution.
- Measurement runner:
  [`scripts/admin-v2-anomalies-perf/measure.mjs`](../scripts/admin-v2-anomalies-perf/measure.mjs)
  - 100 iterations per shape (warmup excluded);
  - the SQL shapes in `measure.mjs` are an inline 1:1 copy of those in
    [`apps/api-v2/src/admin-v2/ledger/anomalies/admin-v2-ledger-anomalies.service.ts`](../apps/api-v2/src/admin-v2/ledger/anomalies/admin-v2-ledger-anomalies.service.ts);
  - one `EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)` captured per shape.
- Environment:
  - PostgreSQL 18.3 (Debian build), local docker container;
  - dev workstation, no concurrent load, warm shared buffers;
  - JIT enabled, default `effective_cache_size`, default
    `random_page_cost`.
- Caveats:
  - this is a synthetic baseline, not production telemetry. It is good
    enough to (a) show order-of-magnitude headroom against the 500 ms
    p95 budget and (b) compare BEFORE vs AFTER for the index reshape
    on the same dataset / same machine.
  - production volumes today are far below 50k (see 3.1a README);
    these numbers should be re-validated once `pg_total_relation_size`
    on `ledger_entry_outcome` crosses the reassessment threshold
    documented in the new migration README.

Raw measurement dumps live under
`scripts/admin-v2-anomalies-perf/output/measure-*.json` (gitignored).

## index reshape rationale

What the BEFORE EXPLAIN told us (full plans below):

- Both `count*` and `list*` LATERAL queries used
  `Index Scan using idx_ledger_entry_outcome_ledger_entry_id_created_at`,
  i.e. the index put in place by
  `20260225140000_ledger_entry_outcome_append_only`.
- That index has key `(ledger_entry_id, created_at DESC)`. It does not
  carry `id` (the secondary sort key) and does not carry `status` (the
  projected column). Consequence: every LATERAL invocation paid for an
  `Incremental Sort` step (`Sort Key: o.created_at DESC, o.id DESC`)
  plus an `Index Scan` (heap-fetching) instead of an `Index Only Scan`.
- Per-loop the cost is small (~3 µs), but it is multiplied by 50k
  outer-loop iterations because the planner picks `Seq Scan` on
  `ledger_entry` (deliberately - the table is small enough that a
  seq scan is the cheapest outer driver). Net effect: ~50k × Sort +
  ~200k buffer hits per probe.

What the new index does:

- `(ledger_entry_id, created_at DESC, id DESC) INCLUDE (status)` -
  exact key match for the LATERAL `ORDER BY ... LIMIT 1`, plus a
  covering INCLUDE so `status` lives inside the index.
- AFTER plans show:
  - `Index Only Scan using idx_ledger_entry_outcome_lateral_covering`
    with `Heap Fetches: 0`;
  - `Incremental Sort` step removed entirely;
  - `Buffers: shared hit` drops from ~200k to ~150k on every probe
    (-25%).
- One composite covering index was enough; we did not need a second
  partial index on pending statuses. The partial-index option was
  rejected because (a) one composite index already moved p95 inside
  the budget with comfortable headroom and (b) maintaining a partial
  index requires keeping its `WHERE status IN (...)` clause in sync
  with `PENDING_OUTCOME_STATUSES` in the service, which is fragile
  for the marginal additional gain.
- The pre-existing `idx_ledger_entry_outcome_ledger_entry_id_created_at`
  is left in place as additive insurance during rollout. Dropping it is
  owned by a future cleanup once production telemetry confirms the
  planner has fully migrated to the covering index.

## Promise.all decision

Cosmetic 3.1a audit observation: `getSummary` runs three count probes
through `Promise.all`. Question: is the parallelism actually pulling
its weight, or could it be reverted to sequential without harm?

Data:

| Shape                       | p50    | p95    | p99    | mean   |
| --------------------------- | -----: | -----: | -----: | -----: |
| summaryEndpoint:promiseAll  | 135 ms | 168 ms | 189 ms | 137 ms |
| summaryEndpoint:sequential  | 252 ms | 383 ms | 455 ms | 263 ms |

`Promise.all` is roughly **2x faster on p95**, and the sequential
variant gets uncomfortably close to the 500 ms budget at p99 (455 ms
on synthetic 50k - that already eats most of the headroom).

**Promise.all decision: keep `Promise.all` in `getSummary`.** The
parallel shape is not a stylistic preference, it is doing measurable
work to keep the summary endpoint inside its p95 budget. Reverting it
would push p99 within ~10% of the failure threshold on synthetic 50k
and likely past it on real production traffic patterns where shared
pool contention and warm-cache assumptions don't hold.

This closes the cosmetic backlog item from the 3.1a audit. The other
two cosmetic items (`ROUTE_TOKENS` for
`admin-v2-ledger-anomalies.controller.ts`, dual empty-state on
[`apps/admin-v2/src/app/(shell)/ledger/anomalies/page.tsx`](../apps/admin-v2/src/app/(shell)/ledger/anomalies/page.tsx))
remain explicit backlog items per the slice plan, owned opportunistic-ally
by the next PR that already touches those files.

## What 3.1c+ should pick up

Not blocked by 3.1b, but worth flagging for the next slice that touches
this surface:

- The outer `Seq Scan on ledger_entry le` is doing 13-15 ms of constant
  work on every count probe, regardless of how the inner LATERAL is
  optimised. At dataset sizes meaningfully above 50k entries the
  planner will switch outer drivers; if it picks the wrong one (e.g.
  picks `ledger_entry_status_created_at_idx` for the outer when status
  filter is on the inner), p95 will regress. Re-run
  `scripts/admin-v2-anomalies-perf/measure.mjs` against the new
  baseline before changing query shape.
- Dropping the dormant 3.1a indexes is an option, but only after a
  measurement window in production confirms they are still unused
  (which they currently are - see 3.1a migration README "Findings from
  query analysis").
- If LATERAL stops being good enough at significantly larger volumes
  (~10x current synthetic), the next escalation is a denormalised
  `ledger_entry.latest_outcome_status` column maintained by an actual
  trigger - which is a slice 3.1c+ migration-safety conversation
  (cited as an explicit escalation trigger in the slice 3.1b plan),
  not a 3.1b change.
